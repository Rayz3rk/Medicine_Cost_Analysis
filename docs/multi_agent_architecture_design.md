# 智能体协作与对话信息共享系统架构设计方案

## 1. 现有智能体能力评估矩阵与缺口分析

目前系统中已定义了三类核心智能体，但在复杂的协同任务中仍存在一定的能力缺口：

| 智能体角色 | 核心能力 | 输入依赖 | 评估得分 | 现有缺口与不足 |
| --- | --- | --- | --- | --- |
| **Cost_Accountant** (成本会计) | 成本精算、占比分析、控制点挖掘 | 生产、运输、储存等结构化成本数据 | 8.5/10 | 缺乏对市场终端价格弹性的感知；单向输出，无法对供应链调整给出二次反馈。 |
| **Pricing_Strategist** (定价策略师) | 利润空间测算、市场定价策略 | 成本基线、产品适应症、市场环境 | 8.0/10 | 缺乏动态博弈能力，未考虑竞品价格和实时医保政策变动。 |
| **Supply_Chain_Manager** (供应链经理) | 仓储优化、运输评估、冷链管理 | 仓储条件、成本数据、包装规格 | 8.0/10 | 无法直接获取会计的成本控制点，导致优化建议可能不符合成本红线。 |

**能力缺口总结：**
1. **缺乏中心路由节点（Orchestrator）**：目前智能体采用线性或独立轮询方式工作（如 `for agent_name in agents_to_run:`），缺乏任务拆解、路由调度和上下文记忆的统一管理者。
2. **缺乏数据清洗与合规智能体（Compliance Agent）**：在流转商业机密（如研发成本真实数据）时，缺乏自动脱敏与审查机制。

## 2. 对话信息共享的必要性论证与隐私合规设计

### 2.1 必要性论证
在药品成本与定价分析中，各环节高度耦合：
* 供应链经理提出的“改用冷链物流”建议，会直接影响成本会计的测算模型；
* 成本会计计算出的新底线，又将直接推翻定价策略师原有的市场策略。
若不实现对话与上下文共享，智能体将陷入信息孤岛，生成的报告会产生逻辑冲突。

### 2.2 隐私合规与数据脱敏设计 (Privacy & Compliance)
在多智能体交互过程中，涉及到企业核心机密（如真实BOM成本）。
* **脱敏规则**：利用正则或NLP实体识别，在共享总线前拦截并替换敏感数值。例如，将“研发成本1.2亿元”模糊化为“高占比研发投入区间（等级A）”。
* **访问控制 (RBAC)**：
  * **Level 1 (内部安全域)**：Orchestrator和Cost_Accountant可获取明细成本。
  * **Level 2 (市场域)**：Pricing_Strategist仅能获取“综合成本底线”和“浮动比例”，无法获取单项设备折旧。

## 3. 基于事件驱动的智能体通信协议

采用基于 **Redis Streams / PubSub** 的事件驱动总线机制，实现各智能体的异步解耦。

### 3.1 协议规范与数据格式 (Data Format)
消息体采用标准的 `CloudEvents` 规范（JSON 格式）：

```json
{
  "specversion": "1.0",
  "id": "msg-uuid-1234",
  "source": "agent:cost_accountant",
  "type": "event.analysis.completed",
  "time": "2026-03-23T10:00:00Z",
  "data_content_type": "application/json",
  "data": {
    "session_id": "session-889",
    "summary": "生产成本占主导，建议优化原材料...",
    "redacted_metrics": {"total_cost_tier": "B"}
  },
  "signature": "hmac-sha256-hash..."
}
```

### 3.2 鉴权机制 (Auth)
* 智能体在接入消息总线前，需通过统一鉴权中心换取 **JWT Token**。
* 发布/订阅消息时，在 Header 中携带 Token，总线验证其 `role` 和 `scope` 权限，防止越权监听（如供应链节点试图订阅原始财务数据通道）。

## 4. 共享对话信息的存储方案

采用多级存储架构，兼顾读写性能与长期知识沉淀：

1. **对话上下文缓存 (Redis)**
   * **作用**：存储当前 Session 的短期记忆，支持快速拉取。
   * **策略**：使用 Redis Hash 结构存储 `session_id` 对应的对话列表，设置 24 小时 TTL 过期。
2. **历史记录持久化与加密 (MongoDB)**
   * **作用**：存储完整对话流、审计日志和决策溯源。
   * **加密**：在应用层对敏感字段（如 `cost_data`）使用 AES-256-GCM 算法加密后存入 MongoDB，实现 At-Rest 静态加密。
3. **知识图谱构建 (Neo4j)**
   * **作用**：从对话中抽取“药品-成分-成本类型-优化建议”的关系三元组，构建图谱，为未来的 RAG 检索提供关联知识支撑。

## 5. 智能体协作流程设计

基于 **Orchestrator-Worker**（中枢调度）模式：

1. **任务分发**：用户发起请求，Orchestrator 接收并拆解任务。首先向 `Cost_Accountant` 发布计算指令。
2. **信息流转**：`Cost_Accountant` 运算完毕，将“脱敏后的成本占比结构”发布到总线。
3. **并行处理**：`Pricing_Strategist` 和 `Supply_Chain_Manager` 监听到成本核算完成事件后，并行启动自身推理。
4. **结果聚合**：Orchestrator 订阅所有 Worker 的完成事件。收集齐三个智能体的结果后，调用大模型进行统一的 `Map-Reduce` 融合总结。
5. **冲突解决机制**：若定价师的“目标售价”低于成本会计的“底线成本”，Orchestrator 会触发 **Conflict_Event**，要求双方开启新一轮“讨价还价”对话（多轮迭代），直至满足数学约束或达到最大迭代次数（如 3 次）。

## 6. 性能监控指标与异常处理策略

### 6.1 核心监控指标 (Metrics)
* **Token Usage**：各智能体消耗的 Prompt/Completion Token 统计（告警阈值设定）。
* **Agent Latency**：单个智能体的推理响应延迟（P95 < 5秒）。
* **Message Throughput**：Redis 消息总线的 QPS。
* **Error Rate**：API 调用失败率及幻觉拦截率。

### 6.2 异常处理策略
* **重试与退避**：当 LLM API 限流或网络抖动时，实施指数退避（Exponential Backoff）重试。
* **死信队列 (DLQ)**：连续 3 次处理失败的消息转移至 Redis 的 Dead Letter Queue，由人工介入排查。
* **断路器 (Circuit Breaker)**：当某一 LLM 节点宕机，自动切换至备用模型（如从 SiliconFlow 降级至本地 Ollama）。

## 7. 灰度发布与回滚方案

* **灰度路由 (Canary Release)**：借助 API 网关，根据请求中的 `Tenant-ID` 或随机 Hash，将 10% 的流量路由到携带新版 Prompt 或新版 LLM 模型的智能体（V2版）。
* **A/B 测试监控**：对比 V1 与 V2 版本的业务验收通过率和计算准确率。
* **一键回滚**：如果 V2 版本的 Error Rate 超过 2% 或出现严重数值幻觉，CI/CD 管道通过切换配置中心（如 Nacos/Consul）的路由规则，在秒级将流量切回 V1。
